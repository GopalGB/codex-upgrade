# gsd-discuss

Between SPEC and PLAN, close the knowledge gaps. Read the repo + PROJECT.md + LESSONS.md first (§F), then ask only the questions whose answers would CHANGE the plan — batch them, don't interrogate. Record the resolved assumptions in `.planning/<phase>/DISCUSS.md` so the plan is built on stated facts, not guesses. Prefer one sharp round over many; address the ambiguous parts you CAN resolve from the code yourself before asking. Adapted from `gsd-discuss-phase`. Pitfall: planning on unvalidated assumptions is the #1 cause of building the wrong thing — but over-asking annoys (auto-execute LAW). The skill is calibration: ask what matters, infer the rest, then commit. Next: `gsd-plan-phase`.

**Tools:** targeted questions, assumption list
