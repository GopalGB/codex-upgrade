# career-coding-interview-execution

The interview scores *signal*, not just a correct answer. Run a fixed loop: **Clarify → Examples → Approach (get buy-in) → Code → Test.** (1) Restate the problem; ask about input size, types, duplicates, empty/null, sorted-ness — constraints reveal the target complexity. (2) Walk a concrete example by hand. (3) State your approach *and complexity* and get a nod *before* coding — this prevents 20 wasted minutes down a wrong path. (4) Code while narrating intent, using clean names and helper functions. (5) Dry-run your own code line-by-line on the example, then enumerate edge cases (empty, single element, overflow, all-same, negatives) and trace them.

**What top candidates do differently:** they treat the interviewer as a teammate — think aloud, take hints gracefully (a hint is a gift, not a failure), and when stuck, state the brute force to get points on the board, then optimize. They write compilable, idiomatic code and self-test before saying 'done.'

**Common mistake:** going silent when stuck (the worst signal — looks like you're frozen), coding before alignment, declaring done without testing, and arguing with a hint. Manage time: if you're 10 min in with no plan, ask for a nudge rather than burning the clock.

**Tools:** UMPIRE/clarify-plan-code-test loop, narration, edge-case enumeration, dry-run trace
